## 2026-04-08 - Pre-calculate invariant values into a map for string operations
**Learning:** Nested loops in API matching logic (like `mergeMatches` and `getApiFirstMatches`) perform heavy regex `.toLowerCase().replace(/[^a-z0-9]/g, '')` repeatedly on the same strings, creating an O(N*M) bottleneck. The codebase memory explicitly mentions: "Performance Preference: For nested loops involving repetitive string operations (like team name normalization with regex), pre-calculate invariant values into a map or a pre-processed array to improve efficiency."
**Action:** Created a memoized `normName` function to cache team name normalizations, converting O(N*M) redundant string regex operations to fast O(1) object property lookups.

## 2024-04-09 - Optimize `levenshtein` distance algorithm space complexity
**Learning:** The previous implementation of the `levenshtein` distance algorithm created a 2D array matrix of size `(N+1) * (M+1)` for each string comparison. Since `levenshtein` is called repeatedly inside nested loops (e.g., `mergeMatches`, checking similarity between all incoming streams and base matches), creating this large matrix on every call causes massive array allocations and triggers expensive garbage collection cycles.
**Action:** Replaced the 2D matrix implementation with a 1D array approach (Wagner-Fischer algorithm maintaining only the previous and current rows, optimized further to a single row + `prev` variable). This reduces space complexity from O(N*M) to O(N), bringing execution time per operation down by ~10x and dramatically lowering GC pressure.
